# /annotate — 현재 파일에 주석 추가

IDE에 현재 열려 있는 파일(또는 `$ARGUMENTS`로 전달된 경로)을 읽고,
아래 주석 규칙에 따라 누락된 주석을 추가합니다.

## 주석 원칙

- 코드만으로 의도가 충분히 드러나는 경우에는 주석을 추가하지 않음
- 구현 자체보다 맥락, 이유, 분기 의도, 비즈니스 의미가 드러나도록 작성
- 기존 주석이 적절하면 유지하고, 중복되거나 의미 없는 주석은 추가하지 않음

## 주석 규칙

### JSDoc (함수 / 훅 / 컴포넌트 최상단)

```ts
/**
 * 한 줄 요약 설명
 *
 * - 세부 동작 설명 1
 * - 세부 동작 설명 2
 *
 * @param paramName 파라미터 설명
 * @returns 반환값 설명
 */
```

- 함수, 훅, 컴포넌트 최상단에 추가
- 컴포넌트는 `@returns`를 생략
- 세부 동작이 자명하면 bullet은 생략 가능
- 내부 보조 함수는 복잡하거나 역할이 바로 드러나지 않는 경우에만 추가

### `//` 인라인·블록 주석 (컴포넌트, 훅 내부)

- 주요 변수, 분기, 계산, 사이드이펙트 앞에 필요한 경우 한 줄 주석을 추가
- 변수명만 풀어쓴 설명은 피하고, 값의 의미나 필요한 맥락을 설명
- WHY가 중요한 로직은 이유가 드러나도록 작성
- 블록 레이블은 실제 동작이나 목적 기준으로 작성
- `// 파라미터 결정`, `// 데이터 처리`처럼 추상적인 단계명만 있는 주석은 사용하지 않음

```ts
// 현재 선택된 필터 값
const selectedCategory = ...
```

- WHY가 필요한 경우 같은 상단에 작성

```ts
// 필수 값이 없으면 요청하지 않음
enabled: !!userId,
```

- 블록 레이블은 내용 기반으로

```ts
// 상세 정보 패널 열기
openDetailPanel(...)
```

- `// 파라미터 결정` — 단계 이름만 있는 추상적 레이블 금지

### `{/* */}` JSX 조건부 렌더링 구분

- 조건부 렌더링의 목적이 바로 드러나지 않거나, 같은 레벨에 여러 분기가 있으면 레이블을 추가
- 레이블은 화면에 보이는 UI의 역할이나 목적 기준으로 작성

```tsx
{
  /* 삭제 확인 다이얼로그 */
}
{
  isDeleteDialogOpen && <DeleteConfirmDialog />;
}
```

## 실행 절차

1. 대상 파일 결정:
   - `$ARGUMENTS`가 있으면 해당 경로 사용
   - 없으면 IDE에서 현재 열린 파일(`ide_opened_file` 컨텍스트) 사용
   - 둘 다 없으면 사용자에게 경로 요청
2. 파일 전체를 읽고 아래 항목을 점검:
   - 함수·훅·컴포넌트에 JSDoc 없는 경우 → 추가
   - 컴포넌트·훅 내부 주요 변수/블록에 `//` 주석 없는 경우 → 추가
   - JSX 조건부 렌더링 분기에 `{/* */}` 없는 경우 → 추가
3. 이미 주석이 적절히 달린 곳은 건드리지 않음
4. 수정된 파일을 직접 저장 (Edit 또는 Write 도구 사용)
5. 추가한 주석 목록을 간략히 요약해 보고
